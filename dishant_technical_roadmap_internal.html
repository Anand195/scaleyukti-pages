Genspark AI — API Reality Check

Full web research on Genspark's actual API capabilities as of April 2026. No sugar-coating. Every team member must read this before starting development.

🚨

Critical Finding: Genspark does NOT offer a public developer API. There is no API key you can just sign up for on their website. Their platform API is enterprise-only — requires direct sales engagement. This is the #1 risk for Phase 1 and must be resolved in Week 1 before a single line of production code is written.

❌

No Public API Signup

As of April 2026, genspark.ai has no self-serve API key portal. API access is available only to Enterprise plan users who contact their sales team directly.

🍪

genspark2api — NOT Production Safe

The open-source wrapper (github.com/deanxv/genspark2api) uses browser session cookies extracted via DevTools. It's reverse-engineered, violates ToS, breaks on ReCaptchaV3, and has no SLA. Do not use in production.

✅

Official Enterprise API

Genspark's enterprise API is OpenAI-compatible (same interface, different base URL + API key). If Dishant has Enterprise/Pro plan, request API credentials from Genspark account team at genspark.ai/business.

🔄

Fallback: OpenAI API Direct

Genspark runs GPT-4.1 and GPT-5-mini under the hood (official OpenAI partnership). If Genspark API isn't available, use OpenAI API directly — same model quality, official API, no risk.

WhatsApp Integration — Technical Spec

Meta's WhatsApp Business Cloud API (not on-premise BSP). Full reference for what Dishant must provide and what we build.

🔑

Required Credentials from Dishant

  • WHATSAPP_PHONE_NUMBER_ID — from Meta Developer Dashboard
  • WHATSAPP_BUSINESS_ACCOUNT_ID — WABA ID
  • META_SYSTEM_USER_TOKEN — permanent token (not user token)
  • WEBHOOK_VERIFY_TOKEN — we define this (random string)
📡

Webhook Architecture

  • Meta POSTs inbound messages to our webhook URL
  • We respond with HTTP 200 immediately (within 3s)
  • Process async — reply via separate POST to Meta API
  • Webhook must be HTTPS with valid SSL cert

Knowledge Base Pipeline
Employeewise RAG — Implementation

Retrieval-Augmented Generation pipeline for Employeewise documentation. This is what makes the bot answer product questions accurately.

📥

Step 1 — Ingest

Load all Employeewise docs (PDF, DOCX, HTML) using Unstructured.io. Clean, normalize, and chunk into 500-token segments with 50-token overlap.

🧮

Step 2 — Embed

Each chunk is converted to a 1536-dim vector using text-embedding-3-small. Stored in Qdrant with metadata: source doc, page, section title.

🔍

Step 3 — Retrieve

On user query, embed the query, find top-5 most similar chunks from Qdrant (cosine similarity). Return context to AI model.

🤖

Step 4 — Generate

Genspark AI receives user question + retrieved context. Generates grounded answer. Cannot hallucinate beyond what docs contain.

System Prompt Template (with RAG context injection)
You are a corporate AI assistant for Dishant's organization.
You help employees with:
- Employeewise software guidance (step-by-step instructions)
- HR queries (leave, attendance, payroll)
- Email drafting and professional writing
- Event ideas and brainstorming
- General productivity support

RULES:
- Always be concise — this is WhatsApp, not email. Keep replies under 300 words.
- For Employeewise questions, ONLY answer from the context provided below.
- If context does not contain the answer, say: "I don't have that information.
  Please contact HR at [contact]."
- Format lists with numbers (1. 2. 3.) or bullets — WhatsApp renders them correctly.
- Never guess policy details. If unsure, escalate to HR.

✅

RAG vs Fine-tuning: We use RAG (not fine-tuning) because Employeewise docs will change over time. RAG lets us update the knowledge base without retraining. Faster, cheaper, more maintainable. Correct approach here.

Risk Register — CTO Assessment
Risks, Cracks & Consequences

Every risk identified. Severity rated. Mitigation plan defined. Team must know all of these before starting.

HIGH Genspark Enterprise API Not Available
Genspark has no public API. If Dishant's account is not Enterprise tier or they can't get API credentials from Genspark, we're blocked immediately. This is the single biggest risk to Phase 1.
✅ Mitigation: Confirm API access in Week 1. If unavailable, switch to OpenAI API directly (GPT-4.1). Same model, full official API, zero delay. Client must be briefed on this possibility upfront.
HIGH Meta Permanent Token Not Set Up Correctly
If Dishant uses a user access token instead of a system user token, it will expire in 60 days and break the integration silently. Has caused major production outages in similar projects.
✅ Mitigation: Walk Dishant through Meta Business Manager → System Users → Create System User → Generate Token with whatsapp_business_messaging permission. Verify token type before deployment.
HIGH WhatsApp 24-Hour Messaging Window
Bot can only send free-form text if employee messaged within the last 24 hours. Outside this window, only approved templates work. If bot tries to initiate conversation or reply after 24h, API returns 131047 error.
✅ Mitigation: This is a reactive bot (responds to employee messages). Design all flows as response-only. Never initiate outbound. If needed in future, create Meta-approved templates.
MEDIUM Employeewise Docs Out of Date / Incomplete
If Dishant provides incomplete docs, the bot will confidently say "I don't have that info" for basic questions. This erodes trust fast.
✅ Mitigation: Before ingestion, audit docs for coverage gaps. Run 20 sample questions against the KB. Identify gaps and ask Dishant to fill them before launch. Set explicit knowledge boundaries in system prompt.
MEDIUM AI Hallucination on Policy Details
Without strict RAG grounding, the model may invent policy details (e.g., leave entitlements). Wrong HR information is a legal/compliance risk.
✅ Mitigation: System prompt explicitly forbids answering outside retrieved context. Add fallback: "For policy confirmation, contact HR at [contact]." All policy answers include source reference.
MEDIUM VPS Downtime = Bot Downtime
n8n is self-hosted on Dishant's VPS. If VPS goes down, the bot goes down. Meta will retry webhooks 3 times; after that, messages are dropped.
✅ Mitigation: Set up UptimeRobot monitoring. Enable n8n process restart via PM2 or Docker restart policy. Document escalation process for Dishant's IT team.
LOW WhatsApp API Rate Limits
WhatsApp Business API has per-number rate limits. For internal corporate use (<1000 employees), this is not a real concern. Becomes relevant if bot is used for mass outreach.
✅ Mitigation: This is internal-only, reactive bot. Rate limits not a concern for Phase 1 scope. Monitor if usage spikes unexpectedly.

Phase 1 — 4 Weeks

WhatsApp AI Agent + Employeewise Knowledge Base

  • W1.1
    Verify Genspark API Access Call Genspark API with Dishant's credentials. Confirm base URL, API key, and model access. If unavailable → activate OpenAI fallback. DO NOT proceed without this confirmed.
  • W1.2
    Meta Developer App Setup Create Meta App → WhatsApp product → configure WABA → generate system user token → test send/receive in Meta's sandbox.
  • W1.3
    n8n Webhook Endpoint Create n8n workflow: Webhook node (GET for verification, POST for messages) → parse payload → extract message body + sender phone → route to AI pipeline.
  • W1.4
    Basic Echo Test End-to-end test: Employee sends "hello" → webhook receives → n8n processes → replies "Hello from bot" via Meta API. Confirm full loop working before adding AI.
  • W2.1
    Employeewise Doc Ingestion Pipeline Collect all docs from Dishant. Process with Unstructured.io. Chunk at 500 tokens / 50 overlap. Embed with text-embedding-3-small. Store in Qdrant collection "employeewise_kb".
  • W2.2
    RAG Retrieval Test Run 30 sample employee questions through Qdrant. Evaluate top-5 chunk relevance. Tune chunk size if needed. Target: >85% relevant retrieval on sample set.
  • W3.1
    AI Integration in n8n Workflow Connect: Inbound message → Qdrant query → inject context → Genspark API call → extract reply → send via Meta API. Full pipeline in n8n with error handling at each step.
  • W3.2
    System Prompt Engineering Finalize system prompt for Dishant's tone, escalation rules, and Employeewise constraints. Test against 50 real employee question scenarios. Iterate until quality is production-grade.
  • W3.3
    Error Handling & Fallbacks Handle: Genspark API timeout → retry 2x → fallback message. Qdrant unavailable → use pure AI without context. Meta API error → log + alert. 3s max response time SLA.
  • W4.1
    Pilot Launch (5–10 Users) Select 5–10 internal test users. Run for 5 business days. Collect feedback. Log all queries + AI responses for quality review. Fix issues before full rollout.
  • W4.2
    Monitoring & Alerting Setup UptimeRobot for webhook endpoint. n8n error notifications to Slack/email. Daily query volume dashboard. Token usage tracking (cost monitoring).
  • W5.1
    Production Rollout Open to all employees. Communicate via internal channels. Provide "how to use" guide. Monitor for first 48 hours closely.
02

Phase 2 — Proposed (Post Phase 1)

Persistent Memory & Context Recall — Research Required Before Scoping

  • P2.1
    Memory Architecture Research Evaluate: Redis conversation history vs PostgreSQL long-term memory vs Mem0 (memory-as-a-service). Decision based on volume and query patterns from Phase 1 data.
  • P2.2
    Per-User Conversation Store Store last N messages per phone number in Redis (short-term, 7-day TTL). Summarise older conversations into PostgreSQL long-term memory table.
  • P2.3
    Memory Injection in Context Window On each message: retrieve user's memory summary + last 5 messages → inject into system prompt alongside RAG context. Model now has full user history awareness.
  • P2.4
    Privacy & Data Governance Define retention policy (30/60/90 days). User opt-out mechanism ("forget my data"). DPDP Act 2023 compliance review for Indian context. Document data handling in ToS.
